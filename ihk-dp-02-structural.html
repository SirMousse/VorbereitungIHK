Adapter Structural
Problem: Inkompatible Schnittstellen verbinden
GoF: Structural #1
Definition (IHK)
Das Adapter-Pattern wandelt die Schnittstelle einer Klasse in eine andere um, die der Aufrufer erwartet. Es ermöglicht die Zusammenarbeit von Klassen, deren Schnittstellen ansonsten inkompatibel wären.
Auch bekannt als: Wrapper-Pattern
Wann einsetzen?
· Fremdbibliothek mit anderem Interface einbinden
· Legacy-Code mit neuem System verbinden
· Drittsystem-API adaptieren (z. B. EU-Stecker → US-Steckdose)
SCHNITTSTELLE EuroStecker: stromEuro() KLASSE USSteckdose: // inkompatibel! stromUS() KLASSE ReiseAdapter IMPL. EuroStecker: privat usGerät: USSteckdose stromEuro(): // übersetzt Euro→US usGerät.stromUS() // Aufrufer nutzt EuroStecker-Interface: adapter = NEU ReiseAdapter(usGerät) adapter.stromEuro() // funktioniert!
Client → [EuroStecker IF] → Adapter → [USSteckdose] ↑ bekannt ↑ übersetzt ↑ fremd
Vor- & Nachteile
✓ Bestehenden Code ohne Änderung wiederverwendbar
✓ Trennt Adaptierungslogik vom Client
✗ Zusätzliche Komplexität durch Zwischenschicht
Eselsbrücke: Adapter
Adapter = Reisestecker — du hast einen Euro-Stecker, brauchst einen US-Anschluss: der Adapter übersetzt dazwischen.
Facade Structural
Problem: Komplexes Subsystem vereinfachen
GoF: Structural #5
Definition (IHK)
Das Facade-Pattern stellt eine vereinfachte Schnittstelle für ein komplexes Subsystem bereit. Der Client muss nur mit der Facade interagieren — die innere Komplexität bleibt verborgen.
Wann einsetzen?
· Bibliothek mit vielen Klassen hat komplexe API
· Schichtenarchitektur: obere Schicht vereinfacht untere
· Einheitlicher Einstiegspunkt für ein Modul
· Heimkino-Fernbedienung: ein Knopf = viele Geräte
// Ohne Facade — komplex: dvd.einlegen() beamer.einschalten() lautsprecher.setzeVolumen(50) licht.dimmen(20) dvd.abspielen() // Mit Facade — einfach: KLASSE HeimkinoFacade: filmSchauen(film): dvd.einlegen(film) beamer.einschalten() lautsprecher.setzeVolumen(50) licht.dimmen(20) dvd.abspielen() kino = NEU HeimkinoFacade() kino.filmSchauen("Matrix") // fertig!
Vor- & Nachteile
✓ Reduziert Abhängigkeiten des Clients
✓ Vereinfachte Nutzung komplexer Systeme
✗ Facade kann zur „Gottklasse" werden
Eselsbrücke: Facade
Facade = Fassade eines Hauses — du siehst nur die schöne Außenseite, nicht die Rohre und Kabel dahinter.
Decorator Structural
Problem: Objekte dynamisch erweitern
GoF: Structural #4
Definition (IHK)
Das Decorator-Pattern fügt einem Objekt dynamisch zur Laufzeit neue Verantwortlichkeiten hinzu, ohne die ursprüngliche Klasse zu verändern. Decorator ist eine Alternative zur Vererbung für die Erweiterung von Funktionalität.
Schlüssel: Decorator implementiert dasselbe Interface wie das dekorierte Objekt.
Wann einsetzen?
· Erweiterung ohne Vererbung (Vererbungshierarchie wäre zu tief)
· Kombinierbare Zusatzfunktionen (Kaffee + Milch + Zucker + Sahne)
· Java I/O-Streams (FileInputStream → BufferedInputStream → DataInputStream)
SCHNITTSTELLE Kaffee: kosten() → Zahl beschreibung() → Text KLASSE EinfacherKaffee IMPL. Kaffee: kosten(): ZURÜCK 1.00 beschreibung(): ZURÜCK "Kaffee" KLASSE MilchDecorator IMPL. Kaffee: privat kaffee: Kaffee // hält Referenz kosten(): ZURÜCK kaffee.kosten() + 0.30 beschreibung(): ZURÜCK kaffee.beschreibung() + "+Milch" // Schichten aufbauen: k = NEU EinfacherKaffee() k = NEU MilchDecorator(k) // 1.30€ k = NEU ZuckerDecorator(k) // 1.50€
Vor- & Nachteile
✓ Flexiblere Alternative zur Vererbung
✓ Kombinierbar zur Laufzeit
✗ Viele kleine Objekte, Debugging schwieriger
Eselsbrücke: Decorator
Decorator = Weihnachtsbaum — Baum bleibt Baum, aber du hängst Kugeln, Lichter, Stern dran. Jede Dekoration fügt etwas hinzu.
Proxy Structural
Problem: Zugriff auf Objekt kontrollieren
GoF: Structural #7
Definition (IHK)
Das Proxy-Pattern stellt einen Stellvertreter für ein anderes Objekt bereit und kontrolliert den Zugriff auf dieses. Der Proxy implementiert dasselbe Interface und kann Zugriffe filtern, cachen, protokollieren oder verzögern.
Proxy-Varianten (IHK)
Virtual Proxy Lazy Loading — Objekt erst bei Bedarf laden
Protection Proxy Zugriffsschutz — Berechtigungen prüfen
Remote Proxy Fernzugriff auf entferntes Objekt (RPC)
Caching Proxy Ergebnisse zwischenspeichern
SCHNITTSTELLE Bild: anzeigen() KLASSE EchteBild IMPL. Bild: EchteBild(datei): ladenVonFestplatte(datei) // teuer! anzeigen(): zeige Bild KLASSE BildProxy IMPL. Bild: privat bild = NULL privat datei anzeigen(): WENN bild == NULL DANN bild = NEU EchteBild(datei) // lazy! bild.anzeigen()
Vor- & Nachteile
✓ Lazy Loading spart Ressourcen
✓ Zugangskontrolle ohne Client-Änderung
✗ Antwort kann verzögert sein
Eselsbrücke: Proxy
Proxy = Assistent/Türsteher — du willst zum Chef (echtes Objekt), aber der Assistent prüft erst, ob du darfst, und besorgt dir die Antwort.
Composite Structural
Problem: Baumstrukturen einheitlich behandeln
GoF: Structural #3
Definition (IHK)
Das Composite-Pattern erlaubt es, Objekte in Baumstrukturen zusammenzusetzen, um Hierarchien zu repräsentieren. Einzelne Objekte (Leaf) und Zusammensetzungen (Composite) werden über dasselbe Interface behandelt — der Client muss nicht unterscheiden.
Wann einsetzen?
· Dateisystem (Datei und Ordner gleich behandeln)
· GUI-Widgets (Button, Panel, Window sind alle „Komponenten")
· Menü-Hierarchien (Menü enthält Menüpunkte oder Untermenüs)
· Unternehmenshierarchie (Mitarbeiter + Abteilung)
SCHNITTSTELLE Komponente: anzeigen() groesse() → Zahl KLASSE Datei IMPL. Komponente: // Leaf anzeigen(): drucke name groesse(): ZURÜCK eigenGroesse KLASSE Ordner IMPL. Komponente: // Composite kinder: Liste von Komponente hinzufügen(k): kinder.add(k) anzeigen(): FÜR ALLE k IN kinder: k.anzeigen() groesse(): ZURÜCK SUMME(k.groesse() FÜR ALLE k)
Komponente (IF) ├── Datei (Leaf) └── Ordner (Composite) ├── Datei └── Ordner └── Datei
Vor- & Nachteile
✓ Einheitliche Behandlung von Einzel- und Gruppenobjekten
✓ Beliebig tiefe Hierarchien
✗ Interface kann zu allgemein werden
Eselsbrücke: Composite
Composite = Dateisystem — Datei und Ordner verhalten sich gleich: beide haben Namen, beide haben Größe. Ordner enthält beides.
Bridge & Flyweight Structural
Bridge — Definition
Das Bridge-Pattern trennt eine Abstraktion von ihrer Implementierung, sodass beide unabhängig voneinander variiert werden können. Statt Vererbung wird Komposition eingesetzt.
Wann: Abstraktion + Implementierung sollen sich unabhängig ändern · Plattformunabhängige GUIs
SCHNITTSTELLE Renderer: render(form) KLASSE VektorRenderer IMPL. Renderer: render(f): zeichne als Vektor KLASSE RasterRenderer IMPL. Renderer: render(f): zeichne als Pixel KLASSE Kreis: // Abstraktion privat renderer: Renderer zeichnen(): renderer.render("Kreis") // Kreis mit Vektor oder Raster: k1 = NEU Kreis(VektorRenderer()) k2 = NEU Kreis(RasterRenderer())
Eselsbrücke: Bridge
Bridge = Brücke zwischen zwei Welten — Form (was) und Renderer (wie) werden getrennt und über eine Brücke verbunden.
Flyweight — Definition (Kurz)
Das Flyweight-Pattern teilt gemeinsame Zustände zwischen vielen feingranularen Objekten, um Speicher zu sparen. Interner (gemeinsamer) Zustand wird geteilt, externer Zustand wird übergeben.
Wann: sehr viele ähnliche Objekte (Buchstaben in Textverarbeitung, Partikel in Spielen)
Eselsbrücke: Flyweight
Flyweight = Schriftart-Cache — 10.000 Buchstaben 'A' teilen sich eine Schriftart-Definition, nur Position ist individuell.

Adapter vs. Facade vs. Proxy — Abgrenzung
Adapter ändert das Interface eines bestehenden Objekts
Facade vereinfacht mehrere Objekte hinter einem Interface
Proxy gleicher Interface wie Original — kontrolliert Zugriff
Decorator gleicher Interface — fügt Funktionalität hinzu
IHK-Falle: Proxy und Decorator ähneln sich — Unterschied: Proxy kontrolliert Zugriff, Decorator erweitert Verhalten!
Entscheidungshilfe
→ Inkompatible Schnittstelle? → Adapter
→ Komplexes Subsystem? → Facade
→ Dynamisch Verhalten hinzufügen? → Decorator
→ Zugriffsschutz / Lazy Loading? → Proxy
→ Baum-Hierarchie? → Composite
→ Abstraktion ≠ Implementierung? → Bridge
