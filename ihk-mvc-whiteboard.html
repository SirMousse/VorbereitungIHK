Was ist MVC? — IHK-Definition
MVC — Model View Controller
MVC ist ein Entwurfsmuster (Design Pattern) der Softwarearchitektur, das eine Anwendung in drei klar voneinander getrennte Schichten unterteilt: Model (Daten & Logik), View (Darstellung) und Controller (Steuerung). Ziel ist die strikte Trennung der Zuständigkeiten (Separation of Concerns).
Entwickelt: 1979 von Trygve Reenskaug (Smalltalk) · Heute Standard in Web- und Desktop-Frameworks
Warum MVC? — Das Problem ohne MVC
Ohne Trennung entsteht Spaghetti-Code: Datenbankzugriff, Berechnung und HTML-Ausgabe stehen in derselben Datei. Änderungen an der Darstellung brechen die Logik — und umgekehrt.
Entwurfsmuster (Design Pattern)
Ein Entwurfsmuster ist eine bewährte, wiederverwendbare Lösung für ein häufig auftretendes Problem im Softwaredesign. Es ist kein fertiger Code, sondern eine Schablone.
MVC gehört zu den „Architekturmustern" — übergeordnet über klassische Design Patterns (GoF)
Kernprinzip: Separation of Concerns (SoC)
Separation of Concerns bedeutet, dass jede Komponente genau eine klar definierte Aufgabe hat und nichts über die internen Details der anderen wissen muss.
Model weiß nicht, wie die Daten dargestellt werden
View weiß nicht, woher die Daten kommen
Controller weiß nicht, wie Daten gespeichert oder dargestellt werden
Eselsbrücke: MVC Grundprinzip
„Mach Vernünftig Code" —
Model = die Masse (Daten & Logik)
View = die Vitrine (Darstellung für den User)
Controller = der Chef (steuert alles)
MVC — Wann einsetzen?
Geeignet wenn…
· Mehrere Darstellungen derselben Daten nötig (Web + App + API)
· Team arbeitet parallel (Frontend / Backend getrennt)
· Anwendung langfristig gewartet und erweitert wird
· Automatische Tests (Unit-Tests) gewünscht sind
· Die Darstellung häufig wechselt (Redesign)
Weniger geeignet wenn…
· Sehr kleine, einmalige Skripte
· Prototypen mit kurzer Lebensdauer
· Anfänger ohne OOP-Kenntnisse (hohe Lernkurve)
Bekannte MVC-Frameworks
Laravel PHP
Django Python (MTV ≈ MVC)
Spring MVC Java

Angular TypeScript (MVVM-Variante)
Model — Daten & Geschäftslogik
IHK-Definition Model
Das Model repräsentiert die Daten der Anwendung und die zugehörige Geschäftslogik. Es ist verantwortlich für den Datenbankzugriff, die Validierung von Daten und die Durchführung von Berechnungen. Das Model hat keinerlei Kenntnis von der Darstellung (View) oder der Steuerung (Controller).
Faustregel: Alles, was mit Daten zu tun hat → gehört ins Model
Aufgaben des Models
· Datenbankzugriff (CRUD: Create, Read, Update, Delete)
· Datenvalidierung (Ist die E-Mail-Adresse gültig?)
· Geschäftsregeln (Rabatt ab 100 € Bestellwert)
· Berechnungen und Datenverarbeitung
· Benachrichtigung des Controllers bei Datenänderung
Was das Model NICHT macht
✗ HTML ausgeben oder formatieren
✗ Benutzereingaben direkt empfangen
✗ Entscheiden, welche Ansicht angezeigt wird
Python — Beispiel Model
class ProduktModel: def __init__(self, db): self.db = db def alle_produkte(self): # Datenbankzugriff hier return self.db.query( "SELECT * FROM produkte") def speichern(self, produkt): # Validierung if produkt['preis'] < 0: raise ValueError("Preis ungültig") self.db.insert(produkt)
Java — Beispiel Model
public class ProduktModel { private List<Produkt> produkte; public List<Produkt> getAlleProdukte() { return produkte; // aus DB laden } public void speichern(Produkt p) { if (p.getPreis() < 0) throw new IllegalArgumentException(); produkte.add(p); } }
Eselsbrücke: Model
Model = Magazin — lagert alle Daten, kennt die Regeln, weiß aber nicht, in welchem Schaufenster (View) sie ausgestellt werden.
View — Darstellung & Benutzeroberfläche
IHK-Definition View
Die View ist ausschließlich für die Darstellung der Daten zuständig. Sie präsentiert dem Benutzer Informationen und nimmt Eingaben entgegen — enthält aber keinerlei Geschäftslogik. Die View erhält Daten vom Controller und stellt sie dar.
Faustregel: Alles, was der Benutzer sieht → gehört in die View
Aufgaben der View
· Daten visuell aufbereiten (HTML, CSS, Templates)
· Formulare und Eingabeelemente anzeigen
· Benutzereingaben an den Controller weiterleiten
· Mehrere Views für dieselben Daten möglich (Web, PDF, JSON)
Was die View NICHT macht
✗ Direkt auf die Datenbank zugreifen
✗ Berechnungen oder Validierungen durchführen
✗ Entscheiden, welche Daten angezeigt werden
Python — Beispiel View (HTML-Template)
# Jinja2 Template (Flask/Django) # produkte.html — NUR Darstellung! <h1>Produktliste</h1> {% for p in produkte %} <p>{{ p.name }} — {{ p.preis }} €</p> {% endfor %} <form action="/kaufen" method="post"> <input name="id" type="text"> <button>Kaufen</button> </form>
Java — Beispiel View (JSP)
// produkte.jsp — NUR Darstellung! <c:forEach var="p" items="${produkte}"> <p>${p.name} — ${p.preis} €</p> </c:forEach>
Eselsbrücke: View
View = Vitrine im Schaufenster — zeigt nur, was der Controller hineingestellt hat. Die Vitrine denkt nicht nach.
Controller — Steuerung & Vermittlung
IHK-Definition Controller
Der Controller ist der Vermittler zwischen Model und View. Er empfängt Benutzereingaben (z. B. HTTP-Requests), ruft die entsprechenden Methoden des Models auf, empfängt die Daten und entscheidet, welche View mit welchen Daten gerendert wird. Der Controller enthält die Anwendungslogik — nicht die Geschäftslogik.
Faustregel: Alles, was zwischen Daten und Darstellung vermittelt → gehört in den Controller
Aufgaben des Controllers
· Benutzereingaben empfangen und interpretieren
· Model-Methoden aufrufen (Daten lesen / schreiben)
· Entscheiden, welche View angezeigt wird
· Daten von Model an View übergeben
· URL-Routing (welche Aktion wird aufgerufen?)
Python — Beispiel Controller (Flask)
from flask import Flask, render_template, request app = Flask(__name__) model = ProduktModel(db) # Route = Eintrittspunkt für Benutzer @app.route('/produkte') def produkte_anzeigen(): daten = model.alle_produkte() # Model return render_template( # View 'produkte.html', produkte=daten) @app.route('/kaufen', methods=['POST']) def kaufen(): id = request.form['id'] model.kauf_verarbeiten(id) return render_template('erfolg.html')
Eselsbrücke: Controller
Controller = Chef im Restaurant — nimmt Bestellung (Eingabe) auf, gibt sie in die Küche (Model), und bringt das fertige Gericht an den Tisch (View).

Eselsbrücke: MVC-Ablauf
„User tippt → Chef nimmt auf → Küche kocht → Chef bringt → Vitrine zeigt"
User → Controller → Model → Controller → View → User
Ohne MVC vs. mit MVC
Daten & HTML
✗ Gemischt in einer Datei
✓ Klar getrennt (Model / View)
Änderung Design
✗ Logik-Code anfassen
✓ Nur View ändern
Logik testen
✗ Browser nötig
✓ Unit-Test auf Model
Team-Arbeit
✗ Konflikte im Code
✓ Frontend / Backend parallel
Vorteile — Stabilität · Wartbarkeit · Flexibilität
Vorteile MVC
Stabilität: Klare Trennung → Änderung in View bricht nie Logik

Wartbarkeit: Jede Schicht isoliert änderbar · Fehler leicht lokalisierbar

Flexibilität: Mehrere Views für ein Model (Web + App + API)

Testbarkeit: Model ohne UI testbar · Unit-Tests möglich

Wiederverwendung: Model in mehreren Projekten nutzbar

Teamarbeit: Frontend & Backend parallel entwickelbar

Lesbarkeit: Klare Struktur → neues Teammitglied findet sich schnell zurecht
Nachteile MVC
Komplexität: Mehr Dateien und Klassen als nötig für kleine Projekte

Lernkurve: Anfänger müssen Konzept verstehen

Overhead: Für sehr einfache Anwendungen überdimensioniert

Enge Bindung: Controller kennt sowohl Model als auch View

Framework-abhängig: Implementierung je Framework verschieden
