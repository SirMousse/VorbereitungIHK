Observer Behavioral
Problem: Abhängige automatisch benachrichtigen
GoF: Behavioral #5
Definition (IHK)
Das Observer-Pattern (Beobachter) definiert eine 1-zu-n-Abhängigkeit zwischen Objekten. Ändert sich das Subject (beobachtetes Objekt), werden alle registrierten Observer (Beobachter) automatisch benachrichtigt. Auch bekannt als: Publish-Subscribe-Muster.
Wann einsetzen?
· MVC: Model benachrichtigt Views bei Datenänderung
· Event-Systeme (Button-Klick → mehrere Handler)
· Nachrichtensysteme (Newsletter, Push-Benachrichtigungen)
· Aktienkurs-Ticker (eine Quelle, viele Anzeigen)
SCHNITTSTELLE Observer: aktualisieren(wert) KLASSE Nachrichtenkanal: // Subject beobachter = [] nachricht = "" anmelden(o): beobachter.hinzufügen(o) abmelden(o): beobachter.entfernen(o) setzeNachricht(n): SELBST.nachricht = n benachrichtigeAlle() benachrichtigeAlle(): FÜR ALLE o IN beobachter: o.aktualisieren(nachricht) KLASSE Abonnent IMPL. Observer: aktualisieren(n): drucke n
Vor- & Nachteile
✓ Lose Kopplung Subject ↔ Observer
✓ Beliebig viele Observer dynamisch
✗ Unerwartete Reihenfolge der Updates
✗ Memory Leaks wenn Observer nicht abgemeldet
Eselsbrücke: Observer
Observer = Newsletter-Abo — du abonnierst, und sobald eine neue Ausgabe erscheint, bekommst du automatisch eine Mail.
Iterator Behavioral
Problem: Sammlung durchlaufen ohne Inneres kennen
Definition (IHK)
Das Iterator-Pattern stellt einen Weg bereit, auf die Elemente einer Sammlung (Collection) sequenziell zuzugreifen, ohne die interne Darstellung (Array, Liste, Baum…) zu kennen oder offenzulegen.
In Java und Python ist Iterator direkt eingebaut (for-each / Iterable).
SCHNITTSTELLE Iterator: hatNächstes() → Wahrheit nächstes() → Element KLASSE ZahlenIterator IMPL. Iterator: privat daten, pos = 0 hatNächstes(): ZURÜCK pos < länge(daten) nächstes(): wert = daten[pos] pos = pos + 1 ZURÜCK wert // Verwendung: iter = NEU ZahlenIterator([1,2,3]) SOLANGE iter.hatNächstes(): drucke iter.nächstes()
Vor- & Nachteile
✓ Client kennt keine Datenstruktur intern
✓ Einheitliches Durchlauf-Interface
✗ Eigener Iterator bei einfachen Listen unnötig
Eselsbrücke: Iterator
Iterator = Fernbedienung für Kanäle — du weißt nicht, wie der TV die Kanäle speichert, aber „nächster Kanal" funktioniert immer.
Strategy Behavioral
Problem: Algorithmus zur Laufzeit austauschen
GoF: Behavioral #9
Definition (IHK)
Das Strategy-Pattern definiert eine Familie von Algorithmen, kapselt jeden einzeln und macht sie austauschbar. Der Algorithmus kann zur Laufzeit gewechselt werden, ohne den Client zu ändern.
Wann einsetzen?
· Sortierverfahren austauschen (Bubble Sort ↔ Merge Sort)
· Zahlungsarten (PayPal, Kreditkarte, Überweisung)
· Kompressionsalgorithmen (ZIP, RAR, GZIP)
· Navigationsrouten (kürzeste, schnellste, günstigste)
SCHNITTSTELLE SortierStrategie: sortiere(daten) KLASSE BubbleSortStrategie IMPL. SortierStrategie: sortiere(daten): // Bubble Sort KLASSE MergeSortStrategie IMPL. SortierStrategie: sortiere(daten): // Merge Sort KLASSE Sortierer: // Context privat strategie: SortierStrategie setzeStrategie(s): SELBST.strategie = s ausführen(daten): strategie.sortiere(daten) // Laufzeit-Wechsel: s = NEU Sortierer() s.setzeStrategie(NEU BubbleSortStrategie()) s.ausführen([3,1,2]) s.setzeStrategie(NEU MergeSortStrategie()) s.ausführen([5,4,6])
Vor- & Nachteile
✓ Algorithmen frei austauschbar
✓ Kein if-else-Wirrwarr im Context
✗ Client muss Strategien kennen
Eselsbrücke: Strategy
Strategy = Navi-App — du wählst: „kürzeste Route", „schnellste Route", „günstigste Route" — der Algorithmus wechselt, das Ziel bleibt gleich.
Command Behavioral
Problem: Aktionen als Objekte kapseln
GoF: Behavioral #2
Definition (IHK)
Das Command-Pattern kapselt eine Anfrage als Objekt. Dadurch können Anfragen parametrisiert, in einer Warteschlange gespeichert, rückgängig gemacht (Undo) oder protokolliert werden.
Wann einsetzen?
· Undo/Redo-Funktion (Texteditor, Grafikprogramm)
· Transaktions-Log (alle Aktionen speichern)
· Aufgabenwarteschlangen (Job Queue)
· GUI-Buttons, Menüpunkte (Befehl entkoppelt von Auslöser)
SCHNITTSTELLE Befehl: ausführen() rückgängig() KLASSE TextEinfügenBefehl IMPL. Befehl: privat editor, text, position ausführen(): editor.einfügen(text, position) rückgängig(): editor.löschen(position, länge(text)) KLASSE BefehlsManager: // Invoker verlauf = [] ausführen(befehl): befehl.ausführen() verlauf.push(befehl) undo(): b = verlauf.pop() b.rückgängig()
Beteiligte Rollen
Invoker löst Befehl aus (Button, Manager)
Command kapselt die Aktion (Interface)
Receiver führt die eigentliche Arbeit aus (Editor)
Client erstellt und konfiguriert den Befehl
Eselsbrücke: Command
Command = Bestellzettel im Restaurant — du schreibst deine Bestellung auf, der Zettel kann weitergegeben, gespeichert, storniert werden.
Template Method Behavioral
Problem: Ablauf fest, Details variabel
GoF: Behavioral #10
Definition (IHK)
Das Template Method-Pattern definiert das Grundgerüst eines Algorithmus in einer Methode der Oberklasse. Einzelne Schritte werden als abstrakte Methoden (Hooks) definiert, die Unterklassen konkret implementieren — das Gerüst selbst bleibt unverändert.
Wann einsetzen?
· Gleicher Ablauf, verschiedene Details (Spielablauf: Runde starten → spielen → Runde beenden)
· Test-Frameworks (setUp → test → tearDown)
· Datenexport (öffne Datei → schreibe Inhalt → schließe Datei)
· Getränkezubereitung (Wasser kochen → aufbrühen → einschenken)
ABSTRAKTE KLASSE GetränkeRezept: // Template Method — fest! zubereiten(): wasserKochen() // fest aufbrühen() // Unterklasse! inTasseGießen() // fest zutatenHinzufügen() // Hook wasserKochen(): drucke "koche Wasser" inTasseGießen(): drucke "gieße ein" ABSTRAKT aufbrühen() zutatenHinzufügen(): // opt. Hook KLASSE Tee ERBT GetränkeRezept: aufbrühen(): drucke "ziehe Teebeutel" KLASSE Kaffee ERBT GetränkeRezept: aufbrühen(): drucke "lasse Kaffee durch" zutatenHinzufügen(): drucke "gib Milch und Zucker"
Vor- & Nachteile
✓ Vermeidet Code-Duplikate im Ablauf
✓ Ablauf zentral kontrollierbar
✗ Unterklassen können nur bestimmte Schritte ändern
✗ Verletzt Liskovsches Substitutionsprinzip ggf.
Eselsbrücke: Template Method
Template Method = Kochrezept-Vorlage — Schritte stehen fest, aber was genau „aufgebrüht" wird, entscheidest du. Rezept = Oberklasse, Getränk = Unterklasse.
State Behavioral
Problem: Verhalten je Zustand ändern
GoF: Behavioral #8
Definition (IHK)
Das State-Pattern ermöglicht es einem Objekt, sein Verhalten abhängig von seinem inneren Zustand zu ändern — als ob das Objekt seine Klasse wechselt. Jeder Zustand wird als eigene Klasse implementiert.
Unterschied zu Strategy: State wechselt selbstständig den Zustand, Strategy wechselt der Client.
Wann einsetzen?
· Automaten (Getränkeautomat, Ampelsteuerung)
· Bestellsystem (neu → bezahlt → versandt → geliefert)
· Verbindungsstatus (verbunden, trennend, getrennt)
· Spielezustände (Menü, Spiel, Pause, Game Over)
SCHNITTSTELLE Zustand: einwerfen() ausgeben() KLASSE LeererZustand IMPL. Zustand: einwerfen(): wechsle zu VollZustand ausgeben(): drucke "leer!" KLASSE VollZustand IMPL. Zustand: einwerfen(): drucke "schon voll!" ausgeben(): wechsle zu LeererZustand KLASSE Automat: // Context privat zustand = NEU LeererZustand() einwerfen(): zustand.einwerfen() ausgeben(): zustand.ausgeben()
Eselsbrücke: State
State = Ampel — dasselbe Objekt, aber je nach Zustand (Rot/Gelb/Grün) anderes Verhalten. Zustand wechselt selbst weiter.

Häufige Verwechslungen (IHK-Falle!)
Strategy vs. State
Strategy Client wechselt Algorithmus bewusst
State Objekt wechselt Zustand selbst — automatisch
Decorator vs. Proxy
Decorator fügt Funktionalität hinzu (Erweiterung)
Proxy kontrolliert Zugriff (Schutz/Lazy Loading)
Observer vs. Command
Observer 1-zu-n, alle werden benachrichtigt
Command 1-zu-1, eine Aktion wird ausgeführt/gespeichert
Template Method vs. Strategy
Template Method Ablauf in Oberklasse fest → Vererbung
Strategy ganzer Algorithmus austauschbar → Komposition
Adapter vs. Facade
Adapter eine Klasse anpassen (inkompatibles Interface)
Facade mehrere Klassen vereinfachen (Subsystem)
