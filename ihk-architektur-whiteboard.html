Client-Server-Architektur — Grundbegriffe
Client-Server-Architektur
Ein Architekturmuster, bei dem Aufgaben zwischen Dienstanforderern (Clients) und Dienstanbietern (Servern) aufgeteilt werden. Die Kommunikation erfolgt über ein Netzwerk nach dem Anfrage-Antwort-Prinzip (Request-Response).

Client
Ein Programm oder Gerät, das Dienste eines Servers in Anspruch nimmt. Der Client sendet Anfragen (Requests) und empfängt Antworten (Responses).
Beispiele: Webbrowser, mobile App, Desktop-Anwendung
Server
Ein System, das Dienste für Clients bereitstellt. Er nimmt Anfragen entgegen, verarbeitet sie und sendet eine Antwort zurück.
Beispiele: Webserver (Apache, nginx), Datenbankserver, Mailserver
Protokoll
Vereinbartes Regelwerk für die Kommunikation zwischen Client und Server.
HTTP/HTTPS für Web · FTP für Dateien · SMTP für E-Mail
Thin Client vs. Fat Client
Thin Client: Minimale lokale Verarbeitung, Logik liegt auf dem Server (z. B. reiner Browser).
Fat Client: Viel Verarbeitung lokal, Server nur für Daten (z. B. Desktop-App).

Schichtenarchitektur (Layered Architecture)
Definition
Die Schichtenarchitektur ist ein Architekturmuster, bei dem eine Anwendung in übereinanderliegende, logisch getrennte Schichten unterteilt wird. Jede Schicht erfüllt eine klar definierte Aufgabe und kommuniziert nur mit der direkt benachbarten Schicht.
3-Schichten-Architektur (3-Tier)
3
Präsentationsschicht
Benutzeroberfläche · Darstellung · HTML, CSS, JS · kein Datenbankzugriff
↕ Nur Schicht-zu-Schicht-Kommunikation
2
Logikschicht
Geschäftslogik · Verarbeitung · Validierung · z. B. Java, Python, PHP
↕ Nur Schicht-zu-Schicht-Kommunikation
1
Datenhaltungsschicht
Datenbankzugriff · Persistenz · SQL · z. B. MySQL, PostgreSQL
Vorteile der Schichtenarchitektur
· Wartbarkeit: Jede Schicht unabhängig änderbar
· Wiederverwendbarkeit: Schichten in anderen Projekten nutzbar
· Testbarkeit: Einzelne Schichten isoliert testbar
· Skalierbarkeit: Schichten unabhängig skalierbar
Nachteile
· Höherer Entwicklungsaufwand
· Leistungsverlust durch Schichtenwechsel
· Strenge Schichtentrennung kann unflexibel sein
2-Tier vs. 3-Tier vs. n-Tier
2-Tier Client + Datenbankserver (keine separate Logikschicht)
3-Tier Client + Applikationsserver + Datenbankserver
n-Tier Beliebig viele Schichten für komplexe Systeme
Fachbegriffe & Prinzipien (IHK-Wissen)
Separation of Concerns (SoC)
Entwurfsprinzip: Verschiedene Zuständigkeiten werden strikt voneinander getrennt. Grundlage für Schichten- und MVC-Architektur.
Kopplung & Kohäsion
Lose Kopplung: Module sind möglichst unabhängig voneinander (Ziel).
Hohe Kohäsion: Ein Modul hat genau eine klar definierte Aufgabe (Ziel).
Schnittstelle (Interface / API)
Definiert, wie zwei Schichten oder Systeme miteinander kommunizieren, ohne interne Implementierungsdetails preiszugeben.
Monolithische Architektur
Gegenteil zur Schichtenarchitektur: Alle Komponenten in einem einzigen, eng verzahnten System. Einfach zu entwickeln, schwer zu warten und skalieren.
MVC — Model View Controller
Definition MVC
MVC ist ein Entwurfsmuster (Design Pattern), das eine Anwendung in drei voneinander getrennte Komponenten aufteilt, um Datenhaltung, Darstellung und Steuerungslogik zu trennen.

⇄

Darstellung
Benutzeroberfläche
HTML / Templates
Model
Das Model repräsentiert die Daten und die Geschäftslogik einer Anwendung. Es ist zuständig für den Datenbankzugriff, die Validierung und die Verarbeitung von Daten. Es hat keine Kenntnis von der Darstellung.
Beispiel: Klasse „Kunde" mit Methoden wie speichern(), laden(), validieren()
View
Die View ist ausschließlich für die Darstellung der Daten zuständig. Sie zeigt dem Benutzer die Informationen an und nimmt Eingaben entgegen, enthält aber keine Geschäftslogik.
Beispiel: HTML-Template, das Kundendaten anzeigt
Controller
Der Controller nimmt Benutzereingaben entgegen, steuert den Ablauf der Anwendung und vermittelt zwischen Model und View. Er ruft Model-Methoden auf und übergibt die Ergebnisse an die View.
Beispiel: HTTP-Request empfangen → Model abfragen → View rendern
Ablauf einer Anfrage im MVC
1. Benutzer sendet Anfrage → Controller empfängt
2. Controller ruft Model auf (Daten holen / verarbeiten)
3. Model gibt Daten zurück an Controller
4. Controller übergibt Daten an View
5. View rendert die Ausgabe für den Benutzer
