Themenbereich: Qualitätssicherung & Softwaretest
Grundbegriffe Testen
1Was versteht man unter einem Softwaretest?
Ein Softwaretest ist die systematische Prüfung einer Software auf Fehler, Abweichungen von der Spezifikation und Qualitätsmängel unter definierten Bedingungen.
Ziel: Fehler finden — nicht beweisen, dass keine Fehler vorhanden sind (Dijkstra).
2Was ist der Unterschied zwischen einem Fehler (Error), einem Defekt (Defect/Bug) und einem Versagen (Failure)?
Error: menschlicher Irrtum beim Entwickeln. Defect/Bug: Fehler im Code (Ursache). Failure: sichtbares Fehlverhalten zur Laufzeit (Auswirkung).
Error → erzeugt → Defect → verursacht → Failure
3Was ist ein Testfall (Test Case)?
Ein Testfall beschreibt eine konkrete Testsituation mit: Eingabedaten, Vorbedingungen, erwarteten Ergebnissen und Nachbedingungen.
Grundlage für reproduzierbare, nachvollziehbare Tests.
4Was ist der Unterschied zwischen Verifikation und Validierung?
Verifikation: „Wird das Produkt richtig gebaut?" (entspricht die Software der Spezifikation). Validierung: „Wird das richtige Produkt gebaut?" (entspricht die Software den Anforderungen des Kunden).
Verifikation = intern · Validierung = kundenbezogen
5Nennen Sie vier Testziele.
Fehler finden, Qualität bewerten, Vertrauen aufbauen, Risiken minimieren.
Tests können Fehlerfreiheit nicht beweisen — nur Fehler aufdecken.
Teststufen (Wann wird getestet?)
6Nennen und erläutern Sie die vier Teststufen.
Unit-Test: einzelne Funktionen/Klassen isoliert testen.
Integrationstest: Zusammenspiel mehrerer Komponenten prüfen.
Systemtest: gesamtes System gegen Anforderungen testen.
Abnahmetest: Kunde prüft, ob Anforderungen erfüllt sind.
Reihenfolge: Unit → Integration → System → Abnahme (von innen nach außen)
7Was ist ein Unit-Test und wer führt ihn durch?
Ein Unit-Test prüft eine einzelne, isolierte Einheit (Funktion, Methode, Klasse) auf korrekte Funktionalität. Durchgeführt vom Entwickler, meist automatisiert.
Werkzeuge: JUnit (Java), pytest (Python), Jest (JavaScript)
8Was ist der Unterschied zwischen Systemtest und Abnahmetest?
Systemtest: intern, durch Tester, prüft technische Anforderungen. Abnahmetest: extern, durch Kunden, prüft ob fachliche Anforderungen erfüllt sind (auch: UAT – User Acceptance Test).
Abnahmetest ist Grundlage für die Abnahme und Bezahlung.
Testmethoden: Black-Box vs. White-Box
9Was versteht man unter Black-Box-Test?
Beim Black-Box-Test wird die Software nur über ihre Schnittstellen getestet, ohne Kenntnis des inneren Aufbaus (Quellcode). Getestet wird: Eingabe → erwartete Ausgabe.
Perspektive des Anwenders. Geeignet für Funktions- und Akzeptanztests.
10Was versteht man unter White-Box-Test (auch: Glass-Box)?
Beim White-Box-Test kennt der Tester den Quellcode und prüft interne Strukturen, Pfade und Verzweigungen gezielt auf Korrektheit.
Perspektive des Entwicklers. Geeignet für Unit- und Integrationstests.
11Was ist der Grey-Box-Test?
Kombination aus Black- und White-Box: Der Tester hat teilweise Kenntnis der internen Struktur (z. B. Datenbankschema), testet aber über die Oberfläche.
Häufig bei Integrationstests eingesetzt.

Testmethoden: Äquivalenzklassen & Grenzwerte
12Was ist die Äquivalenzklassenanalyse?
Die Äquivalenzklassenanalyse teilt mögliche Eingabewerte in Gruppen (Klassen), die sich gleich verhalten. Pro Klasse wird nur ein repräsentativer Wert getestet, um die Anzahl der Testfälle zu reduzieren.
Gültige Klassen (EC+) und ungültige Klassen (EC−) werden unterschieden.
13Erklären Sie die Grenzwertanalyse anhand eines Beispiels.
Die Grenzwertanalyse testet Werte an den Rändern der Äquivalenzklassen, da dort besonders häufig Fehler auftreten. Beispiel: Alter 0–120 → Testfälle: −1, 0, 1, 119, 120, 121.
Ergänzt die Äquivalenzklassenanalyse — beide gemeinsam verwenden!
14Ein Eingabefeld akzeptiert Zahlen von 1 bis 100. Nennen Sie typische Testfälle.
Gültig: 1, 50, 100. Ungültig: 0, −1, 101, 200. Grenzwerte: 0, 1, 100, 101. Sonderfälle: leer, Buchstabe, Kommazahl.
IHK erwartet: gültige + ungültige Klassen + Grenzwerte benennen.
Weitere Testarten
15Was ist ein Regressionstest?
Ein Regressionstest prüft nach einer Änderung (Bugfix, neues Feature), ob bisher funktionierende Teile der Software noch korrekt arbeiten.
Verhindert, dass Änderungen neue Fehler (Regressionsfehler) einführen.
16Was ist der Unterschied zwischen manuellem und automatisiertem Testen?
Manuell: Tester führt Tests von Hand aus — flexibel, teuer, fehleranfällig. Automatisiert: Testskripte laufen automatisch — schnell, wiederholbar, hohe Anfangsinvestition.
Automatisierung lohnt sich besonders für Regressionstests.
17Was ist ein Lasttest / Stresstest?
Lasttest: prüft das Verhalten unter erwarteter Last (z. B. 1000 gleichzeitige Nutzer). Stresstest: prüft das Verhalten unter Überlast bis zum Systemversagen.
Beide gehören zu den nicht-funktionalen Tests.
Statische vs. Dynamische Tests
18Was ist der Unterschied zwischen statischem und dynamischem Test?
Statischer Test: Software wird geprüft, ohne sie auszuführen (z. B. Code-Review, Inspektion). Dynamischer Test: Software wird ausgeführt und das Verhalten geprüft.
Statisch = früh & günstig · Dynamisch = Laufzeitverhalten
19Nennen Sie drei statische Testmethoden.
Code-Review (Vier-Augen-Prinzip), Walkthrough (Autor erklärt Code), Inspektion (formaler Prozess mit Rollen), statische Code-Analyse (automatisiert, z. B. SonarQube).
Ohne Programmausführung — Fehler frühzeitig und kostengünstig finden.
20Was versteht man unter Testabdeckung (Code Coverage)?
Die Testabdeckung gibt an, welcher Anteil des Quellcodes durch Tests ausgeführt wird. Typen: Anweisungsabdeckung, Zweigabdeckung, Pfadabdeckung.
100 % Coverage bedeutet nicht fehlerfreie Software — nur: jede Zeile wurde durchlaufen.
21Was ist der Unterschied zwischen Anweisungs- und Zweigabdeckung?
Anweisungsabdeckung (C0): jede Anweisung mindestens einmal ausgeführt. Zweigabdeckung (C1): jeder Zweig (if/else, true/false) mindestens einmal ausgeführt. C1 ist stärker als C0.
IHK-relevant: C1 (Zweigabdeckung) ist Mindeststandard für sinnvolle Tests.
Testprozess & Qualitätssicherung
22Nennen Sie die Phasen des Testprozesses.
1. Testplanung, 2. Testanalyse & Testentwurf, 3. Testrealisierung, 4. Testdurchführung, 5. Testauswertung & Abschlussbericht.
Analog zum Softwareentwicklungsprozess — Test beginnt nicht erst beim fertigen Produkt.
23Was ist der Unterschied zwischen Qualitätssicherung (QS) und Qualitätskontrolle (QK)?
QS (Quality Assurance): proaktiv — Prozesse und Standards einhalten, um Fehler zu vermeiden. QK (Quality Control): reaktiv — fertiges Produkt prüfen und Fehler finden.
Testen ist Teil der Qualitätskontrolle; Code-Reviews Teil der Qualitätssicherung.
24Was ist Test-Driven Development (TDD)?
Bei TDD werden Tests vor dem eigentlichen Code geschrieben. Zyklus: Test schreiben (schlägt fehl) → Code schreiben (Test besteht) → Refactoring → wiederholen.
Vorteil: Sauberes Design, hohe Testabdeckung von Anfang an (Red-Green-Refactor).
25Nennen Sie drei Merkmale von Softwarequalität nach ISO 25010.
Funktionalität, Zuverlässigkeit, Benutzbarkeit (Usability), Effizienz (Performance), Wartbarkeit, Übertragbarkeit (Portabilität), Sicherheit.
ISO 25010 ersetzt ISO 9126 — Qualität umfasst mehr als nur Fehlerfreiheit.
