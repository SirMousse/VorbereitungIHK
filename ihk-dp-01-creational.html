Design Patterns — Grundlagen & Einteilung
Was ist ein Design Pattern?
Ein Design Pattern (Entwurfsmuster) ist eine bewährte, wiederverwendbare Schablone zur Lösung häufig wiederkehrender Entwurfsprobleme in der Softwareentwicklung. Es beschreibt ein allgemeines Problem und eine erprobte Lösung — unabhängig von der Programmiersprache.
Ursprung: „Gang of Four" (GoF) — Gamma, Helm, Johnson, Vlissides (1994): „Design Patterns: Elements of Reusable Object-Oriented Software"

Singleton Creational
Problem: Nur eine Instanz erlaubt
GoF: Creational #1
Definition (IHK)
Das Singleton-Pattern stellt sicher, dass von einer Klasse genau eine Instanz existiert und bietet einen globalen Zugriffspunkt auf diese Instanz. Der Konstruktor ist privat — Erzeugung nur über eine Klassenmethode möglich.
Wann einsetzen?
· Datenbankverbindung (nur eine nötig)
· Logger (zentrales Protokoll)
· Konfigurationsobjekt (einmalig geladen)
· Druckerspooler (nur eine Warteschlange)
KLASSE Singleton: PRIVAT STATISCH instanz = NULL PRIVAT Konstruktor() // gesperrt! ÖFFENTLICH STATISCH gibInstanz(): WENN instanz == NULL DANN instanz = NEU Singleton() ZURÜCK instanz
Vor- & Nachteile
✓ Genau eine Instanz garantiert
✓ Globaler Zugriff
✗ Schwer testbar (globaler Zustand)
✗ Verletzt Single-Responsibility-Prinzip
Eselsbrücke: Singleton
Singleton = Einzelkind — es gibt immer nur genau eines, egal wie oft man fragt.
Factory Method Creational
Problem: Objekterzeugung delegieren
GoF: Creational #3
Definition (IHK)
Die Factory Method definiert eine Schnittstelle zur Erzeugung eines Objekts, überlässt aber den Unterklassen die Entscheidung, welche konkrete Klasse instanziiert wird. Die Erzeugungslogik ist vom Aufrufer entkoppelt.
Wann einsetzen?
· Wenn der Typ des Objekts zur Laufzeit bestimmt wird
· Wenn Unterklassen die Kontrolle über Erzeugung haben sollen
· Erstellung von Dokumenten (PDF, DOCX, HTML je nach Typ)
ABSTRAKTE KLASSE Fahrzeugfabrik: ABSTRAKT erstelleFahrzeug() → Fahrzeug KLASSE AutoFabrik ERBT Fahrzeugfabrik: erstelleFahrzeug(): ZURÜCK NEU Auto() KLASSE LkwFabrik ERBT Fahrzeugfabrik: erstelleFahrzeug(): ZURÜCK NEU Lkw() // Aufrufer kennt nur "Fahrzeugfabrik" fabrik = NEU AutoFabrik() fzg = fabrik.erstelleFahrzeug()
Vor- & Nachteile
✓ Aufrufer vom konkreten Typ entkoppelt
✓ Neue Typen ohne Änderung des Aufrufers
✗ Viele Unterklassen nötig
✗ Komplexer als direktes new
Fahrzeugfabrik (abstrakt) └── AutoFabrik → erstellt: Auto └── LkwFabrik → erstellt: Lkw
Eselsbrücke: Factory Method
Fabrik = jemand anderes baut — du bestellst ein „Fahrzeug", welches genau gebaut wird, entscheidet die Fabrik.
Abstract Factory Creational
Problem: Familien kompatibler Objekte
GoF: Creational #1
Definition (IHK)
Die Abstract Factory stellt eine Schnittstelle zur Erzeugung von Familien zusammengehöriger Objekte bereit, ohne deren konkrete Klassen zu benennen. Alle erzeugten Objekte passen zueinander.
Wann einsetzen?
· GUI-Elemente für verschiedene Betriebssysteme (Windows-Button + Windows-Textfeld passen zusammen)
· Datenbank-Treiber (MySQL- oder Oracle-Familie)
· Theme-System (Dark-Mode vs. Light-Mode)
SCHNITTSTELLE GUIFabrik: erstelleButton() → Button erstelleTextfeld() → Textfeld KLASSE WindowsFabrik IMPL. GUIFabrik: erstelleButton() → WinButton erstelleTextfeld() → WinTextfeld KLASSE MacFabrik IMPL. GUIFabrik: erstelleButton() → MacButton erstelleTextfeld() → MacTextfeld
Unterschied Factory Method vs. Abstract Factory
Factory Method erzeugt ein Produkt
Abstract Factory erzeugt eine Familie zusammenpassender Produkte
Eselsbrücke: Abstract Factory
Abstract Factory = IKEA-Kollektion — alle Möbel einer Serie passen zusammen. Wechsle die Fabrik → alles wechselt.
Builder Creational
Problem: Komplexe Objekte schrittweise bauen
GoF: Creational #2
Definition (IHK)
Das Builder-Pattern trennt die Konstruktion eines komplexen Objekts von seiner Repräsentation. Es erlaubt, dasselbe Konstruktionsverfahren für verschiedene Darstellungen zu verwenden. Der Director steuert den Bauprozess, der Builder führt ihn aus.
Wann einsetzen?
· Objekte mit vielen optionalen Parametern (Pizza-Builder)
· Dokumente mit verschiedenen Formaten (HTML, PDF, Markdown)
· SQL-Query-Builder
· Konfigurationsobjekte
KLASSE PizzaBuilder: setzeTeig(typ): SELBST.teig = typ ZURÜCK SELBST // Method Chaining setzeBelag(belag): SELBST.belag = belag ZURÜCK SELBST setzeGroesse(gr): SELBST.groesse = gr ZURÜCK SELBST baue(): ZURÜCK Pizza(SELBST) // Verwendung mit Method Chaining: pizza = PizzaBuilder() .setzeTeig("dünn") .setzeBelag("Salami") .setzeGroesse("L") .baue()
Beteiligte Rollen
Builder definiert Bauschritte (Interface)
ConcreteBuilder implementiert Bauschritte
Director steuert die Reihenfolge
Product das fertige komplexe Objekt
Vor- & Nachteile
✓ Schrittweise, lesbare Objektkonstruktion
✓ Verschiedene Repräsentationen möglich
✗ Mehr Klassen als nötig für einfache Objekte
Eselsbrücke: Builder
Builder = Burger-King-Bestellung — Schritt für Schritt: erst Brötchen, dann Fleisch, dann Belag. Am Ende: fertiger Burger.
Prototype Creational
Problem: Objekte effizient kopieren
GoF: Creational #4
Definition (IHK)
Das Prototype-Pattern ermöglicht das Erstellen neuer Objekte durch Kopieren (Klonen) eines bestehenden Objekts (des Prototyps). Statt mit new zu erzeugen, wird ein vorhandenes Objekt dupliziert.
Wann einsetzen?
· Objekterzeugung ist teuer (DB-Zugriff, Berechnung)
· Viele ähnliche Objekte werden benötigt
· Klassen zur Laufzeit unbekannt
· Spielfiguren mit gleichen Grundwerten klonen
SCHNITTSTELLE Klonbar: klone() → Klonbar KLASSE Spielfigur IMPL. Klonbar: name, leben, staerke klone(): // Flache Kopie des Objekts ZURÜCK NEU Spielfigur(SELBST) // Verwendung: original = NEU Spielfigur("Held", 100, 50) kopie1 = original.klone() kopie2 = original.klone() // kopie1 und kopie2 unabhängig von original
Flache vs. tiefe Kopie
Flache Kopie kopiert Referenzen (Objekte geteilt)
Tiefe Kopie kopiert alle Objekte rekursiv (völlig unabhängig)
IHK kann nach dem Unterschied fragen!
Vor- & Nachteile
✓ Schneller als neue Erzeugung
✓ Klasse muss nicht bekannt sein
✗ Tiefe Kopie aufwendig bei Zirkelreferenzen
Eselsbrücke: Prototype
Prototype = Kopiermaschine — statt neu bauen: einmal einscannen, beliebig oft ausdrucken.

Entscheidungshilfe: Welches Pattern wann?
→ Nur eine Instanz erlaubt? → Singleton
→ Welches Objekt erst zur Laufzeit klar? → Factory Method
→ Ganze Familie kompatibler Objekte? → Abstract Factory
→ Objekt hat viele optionale Teile? → Builder
→ Neues Objekt aus bestehendem ableiten? → Prototype
Gemeinsames Ziel aller Erzeugungsmuster
Erzeugungsmuster kapseln die Logik der Objekterzeugung. Der Aufrufer wird von den Details der Erzeugung entkoppelt — er weiß nicht, welche konkrete Klasse instanziiert wird.

SOLID-Prinzipien (IHK-Kontext)
Was sind SOLID-Prinzipien?
SOLID sind fünf grundlegende Entwurfsprinzipien der OOP, die guten, wartbaren Code fördern. Design Patterns folgen SOLID-Prinzipien.
BuchstabePrinzipBedeutung
SSingle ResponsibilityKlasse hat genau eine Aufgabe
OOpen/ClosedOffen für Erweiterung, geschlossen für Änderung
LLiskov SubstitutionUnterklasse ersetzt Oberklasse ohne Fehler
IInterface SegregationViele kleine statt einem großen Interface
DDependency InversionAbhängig von Abstraktion, nicht Implementierung
Eselsbrücke: SOLID
Single · Offen · Liskov · Interface · Dependency — „Sehr Oft Lernen Ist Dumm" 😄
